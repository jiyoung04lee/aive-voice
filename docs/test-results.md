## 3. 인증 토큰 처리

실제 전사 과정에서 다음 동작을 확인했다.

- 최초 요청 시 새 인증 토큰 발급
- 이후 폴링에서는 캐시된 토큰 재사용
- 강제 HTTP 401 발생 시 토큰 캐시 초기화
- 새 토큰 발급 후 POST 요청 1회 재시도
- 재시도 결과 HTTP 201, 이후 상태 조회 HTTP 200
- 테스트용 환경변수와 임시 코드는 검증 후 제거

토큰 캐시는 현재 Node.js 프로세스 안에서만 유지된다. 서버 재시작 또는 다중 인스턴스 환경에서는 각 프로세스가 별도 캐시를 사용한다.

## 4. 키워드 부스팅 비교

동일한 음성과 STT 설정에서 키워드 입력 전후 결과를 비교했다.

| 키워드 | 부스팅 전 | 부스팅 후 | 판단 |
|---|---|---|---|
| 에이아이빅데이터융합경영학과 | 일부 구간에서 `AI 빅데이터 융합결여학과` | `AI 빅데이터 융합경영학과` | 개선 |
| 데이터사이언스융합전공 | `데이터 사이언스 융합 정봉도` | `데이터 사이언스 융합 전공도` | 개선 |
| 에이브 | `에이브` | `에이브` | 변화 없음 |
| 현대오토에버 | `현대 오토웨버`, `현대 오토예버` 등 | `현대오토에버`, `현대 오토에버` | 개선 |
| 카카오뱅크 | `카카오뱅크` | `카카오뱅크` | 변화 없음 |
| 데이터엔지니어 | `데이터 엔지니어` | `데이터 엔지니어` | 변화 없음 |
| 프론트엔드개발자 | `프론트엔드 개발자`, 일부 `프론트 핸드` | 동일 | 변화 없음 |

7개 키워드 중 학과명, 전공명, `현대오토에버`에서 개선이 확인됐다. 나머지는 부스팅 전부터 정확했거나 오인식이 유지됐다.

키워드는 인식 정확도를 보조하지만 입력한 띄어쓰기나 영문 표기를 그대로 강제하지 않았다. 또한 `프론트 핸드`, `데이터 시 바화`, `커피책` 등의 오류는 남아 있었다. 한 개의 테스트 음성 결과이므로 일반적인 성능 향상으로 확대 해석하지 않는다.

## 5. 개인정보 표시 마스킹

| 유형 | RTZR 전사 결과 | 마스킹 ON | 판단 |
|---|---|---|---|
| 전화번호 | `전화나 문자는 01012345678이고` | `전화나 문자는 [전화번호]이고` | 정상 |
| 구어체 이메일 | `시니어 앳 에이브 닷 컴 앤드로` | `[이메일]로` | 정상 |

마스킹 OFF에서는 원본 전사문이 표시됐으며, 마스킹 ON에서는 `[전화번호]`, `[이메일]`을 기준으로 검색할 수 있었다.

이 기능은 화면과 TXT 생성 단계에서만 적용되는 표시용 후처리다. 원본 RTZR 응답을 삭제하지 않으며 이름, 주소, 주민등록번호, 계좌번호 등은 처리하지 않는다.

## 8. 결론

로컬 환경에서 파일 업로드부터 전사, 검색, 마스킹, TXT 다운로드, 오디오 재생과 발화 동기화까지의 핵심 흐름이 정상 동작했다.

인증 토큰 캐싱과 401 재시도도 실제로 확인했으며, 키워드 부스팅은 일부 고유명사와 전공명 인식 개선에 도움을 줬다. 다만 모든 오류를 교정하지는 않았고, 개인정보 마스킹 역시 제한된 패턴을 처리하는 표시용 기능이다.

## 3. 인증 토큰 처리

실제 전사 과정에서 다음 동작을 확인했다.

- 최초 요청 시 새 인증 토큰 발급
- 이후 폴링에서 캐시된 토큰 재사용
- 강제 HTTP 401 발생 시 토큰 캐시 초기화
- 새 토큰 발급 후 요청 1회 재시도
- 재시도 결과 POST HTTP 201, 이후 상태 조회 HTTP 200
- 테스트용 환경변수와 임시 코드 제거

토큰 캐시는 현재 Node.js 프로세스 안에서만 유지된다. 서버 재시작 또는 다중 인스턴스 환경에서는 각 프로세스가 별도 캐시를 사용한다.

## 4. 키워드 부스팅 비교

동일한 음성과 STT 설정에서 다음 키워드를 입력 전후로 비교했다.

```text
에이아이빅데이터융합경영학과,
데이터사이언스융합전공,
현대오토에버,
프론트엔드,
카카오뱅크,
커피챗
```

| 키워드 | 적용 전 | 적용 후 | 판단 |
|---|---|---|---|
| 에이아이빅데이터융합경영학과 | `AI 빅데이터 융합경영학과` | 동일 | 변화 없음 |
| 데이터사이언스융합전공 | `데이터 사이언스 융합 전공` | 동일 | 변화 없음 |
| 현대오토에버 | `현대오타에버`, `현대우토예버` | `현대오토에버`, `현대 오토에버` | 개선 |
| 프론트엔드 | `프론텐드`, `프론트 핸드` | `프론트엔드`, `프론트 엔드`, 일부 `프론텐드` | 개선 |
| 카카오뱅크 | `카카오뱅크` | 동일 | 변화 없음 |
| 커피챗 | `커피 탓` | `커피챗` | 개선 |

6개 키워드 중 3개에서 개선이 확인됐고, 3개는 기존 결과를 유지했다. 명확하게 악화된 대상 키워드는 확인되지 않았다.

키워드 부스팅은 특정 고유명사의 인식을 보조했지만 모든 오류를 교정하지는 않았다. `프론텐드`와 같은 일부 오류와 일반 문맥의 오인식은 남아 있어 최종 대화록에는 사람의 검수가 필요하다.

한 개의 테스트 음성 결과이므로 일반적인 성능 향상으로 확대 해석하지 않는다.

## 5. 개인정보 표시 마스킹

| 유형 | 마스킹 OFF | 마스킹 ON | 판단 |
|---|---|---|---|
| 전화번호 | `01012345678` | `[전화번호]` | 정상 |
| 구어체 이메일 | `시니어 앳 AB 점 컴으로` | `[이메일]로` | 정상 |

마스킹 ON에서는 `[전화번호]`와 `[이메일]`이 화면과 TXT에 모두 반영됐다. 마스킹 OFF에서는 원본 전사 문자열을 검색할 수 있었다.

구어체 이메일의 도메인이 영문으로 전사되는 사례를 반영해 정규식을 보완했다.

이 기능은 화면과 TXT 생성 단계에서만 적용되는 표시용 후처리다. 원본 RTZR 응답을 삭제하지 않으며 이름, 주소, 주민등록번호, 계좌번호 등은 처리하지 않는다.

## 6. 오류 처리

| 오류 케이스 | 실제 결과 | 판단 |
|---|---|---|
| 지원하지 않는 파일 | 지원 형식 안내 문구 표시, 전사 요청 미실행 | 통과 |
| 환경변수 누락 | 설정 미완료 안내 표시, 인증 정보 미노출 | 통과 |
| 0바이트 파일 | 이번 최종 테스트에서 실행하지 않음 | 미실행 |

## 8. 결론

로컬 환경에서 음성 업로드, 전용 로딩 화면, 비동기 상태 조회, 화자 분리, 연속 발화 그룹화, 화자 이름 표시, 대화 검색, 개인정보 표시 마스킹, TXT 다운로드까지의 핵심 흐름이 정상 동작했다.

키워드 부스팅은 `현대오토에버`, `프론트엔드`, `커피챗` 인식 개선에 도움을 줬다. 전화번호와 영문 도메인이 포함된 구어체 이메일도 표시 단계에서 정상적으로 마스킹됐다.

다만 키워드 부스팅과 정규식 마스킹 모두 제한적인 보조 기능이며, 실제 게시나 저장 전에 전사 결과를 사람이 검수해야 한다.

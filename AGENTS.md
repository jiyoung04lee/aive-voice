# AIVE Voice

## 프로젝트 개요

AIVE Voice는 RTZR File STT OpenAPI를 활용해 선배 인터뷰 음성을 전사하는 로컬 실행 기반 Next.js 프로토타입이다.

주요 기능:

- 음성 파일 업로드
- 서버에서 RTZR 인증
- 비동기 전사 요청
- 5초 간격 상태 조회
- 화자별 발화 및 타임스탬프 표시
- 부스팅 키워드 입력
- 전사문 검색 및 하이라이트
- TXT 다운로드
- 선택적 전화번호·이메일 표시 마스킹

## 기술 스택

- Next.js App Router
- TypeScript
- React
- Tailwind CSS
- Next.js Route Handler
- Fetch API와 FormData
- npm

FastAPI, Axios, 데이터베이스, 로그인, 상태 관리 라이브러리는 추가하지 않는다.

## 구현 규칙

- RTZR API 관련 로직은 `lib/rtzr.ts`에 작성한다.
- Route Handler는 요청 검증, `lib/rtzr.ts` 함수 호출, JSON 응답 반환만 담당한다.
- RTZR API 주소, 요청 필드, 응답 필드, 설정값을 추측하지 않는다.
- RTZR 공식 문서와 실제 응답을 기준으로 구현한다.
- 전체 앱을 한 번에 구현하지 않고 작은 작업 단위로 진행한다.
- P0가 정상 동작하기 전에는 P1 기능을 구현하지 않는다.
- 새로운 패키지를 임의로 설치하지 않는다.
- 현재 Next.js 설정을 불필요하게 변경하지 않는다.
- 화면 문구와 사용자 오류 메시지는 한국어로 작성한다.
- 불필요한 추상화와 과도한 리팩터링을 하지 않는다.
- 수정 요청과 관련 없는 파일은 변경하지 않는다.

## 보안 규칙

- `RTZR_CLIENT_ID`와 `RTZR_CLIENT_SECRET`은 서버에서만 사용한다.
- RTZR 인증 정보에 `NEXT_PUBLIC_` 접두사를 사용하지 않는다.
- 인증 정보는 서버의 `process.env`에서만 읽는다.
- `.env.local`의 내용을 출력하거나 임의로 수정하지 않는다.
- 실제 인증 정보를 코드, 로그, 문서에 작성하지 않는다.
- `.env.example`에는 환경변수 이름만 작성한다.
- 실제 개인정보와 실제 선배 인터뷰 음성을 사용하지 않는다.
- 테스트에는 가상 정보만 사용한다.
- 인증 정보를 클라이언트 응답에 포함하지 않는다.

## RTZR 구현 규칙

- 전사 요청 후 작업 ID를 반환한다.
- 클라이언트에서 5초 간격으로 전사 상태를 조회한다.
- 완료, 실패, 타임아웃 또는 복구 불가능한 오류가 발생하면 폴링을 중단한다.
- 실제 진행률이 없으므로 임의의 진행률 퍼센트를 표시하지 않는다.
- 원본 RTZR 응답은 유지하고 화면에 필요한 필드만 별도로 변환한다.
- 외부 응답 구조를 확인하기 전에는 확정되지 않은 타입을 임의로 만들지 않는다.
- 실제 응답을 확인한 뒤 애플리케이션에 필요한 TypeScript 타입을 정의한다.
- 키워드가 비어 있으면 RTZR 요청 config에 `keywords`를 포함하지 않는다.

## 기능 우선순위

### P0 — 핵심 STT

1. RTZR 인증
2. 음성 파일 업로드
3. 파일 전사 요청
4. 작업 ID 반환
5. 전사 상태 조회
6. 완료 및 실패 처리
7. 화자별 발화 표시
8. 타임스탬프 표시
9. 기본 오류 처리

### P1 — 활용 기능

1. 부스팅 키워드 입력
2. 전사문 검색
3. 검색어 하이라이트
4. 검색 결과 개수 표시
5. TXT 다운로드

### P1.5 — 조건부 기능

1. 전화번호·표준 이메일 표시 마스킹
2. 마스킹 on/off 토글
3. 타임스탬프 클릭 시 오디오 위치 이동

P0가 정상 동작하기 전에는 P1을 구현하지 않는다.

P0, P1, 문서 작성, 빌드 검증이 완료되지 않았다면 P1.5를 시작하지 않는다.

## 코딩 규칙

- TypeScript 타입을 명확하게 작성한다.
- `any` 사용을 최소화한다.
- 실제 외부 응답 구조를 확인하기 전에는 필요한 최소 범위에서만 임시 타입을 사용한다.
- 작고 단순한 함수로 구현한다.
- 브라우저와 Node.js 기본 API를 우선 사용한다.
- Fetch API와 FormData를 사용한다.
- 클라이언트 컴포넌트는 필요한 파일에만 `"use client"`를 선언한다.
- 서버 전용 코드를 클라이언트 컴포넌트에서 import하지 않는다.
- 오류는 내부 로그용 메시지와 사용자 표시용 메시지를 구분한다.
- 비개발자도 이해할 수 있는 오류 문구를 제공한다.

## 개인정보 표시 마스킹 규칙

- 마스킹 로직은 `lib/masking.ts`에 순수 함수로 작성한다.
- 전화번호와 표준 이메일 주소만 처리한다.
- 이름, 주소, 계좌번호 등 문맥 판단이 필요한 정보는 탐지하지 않는다.
- 원본 RTZR 응답은 수정하지 않는다.
- 화면 표시와 TXT 다운로드 시 선택적으로 마스킹한다.
- 클라이언트 표시 마스킹을 완전한 개인정보 비식별화 기능으로 표현하지 않는다.
- 실제 STT 출력 형태를 확인한 뒤 정규식을 조정한다.

## 작업 방식

- 작업을 시작하기 전에 변경할 파일과 구현 계획을 먼저 설명한다.
- 한 번에 하나의 작은 작업만 수행한다.
- 전체 애플리케이션을 한 번에 구현하지 않는다.
- 확정된 기능 범위 밖의 기능을 추가하지 않는다.
- 새로운 프로덕션 의존성을 설치하기 전에 사용자에게 확인한다.
- 중요한 구현 결정은 면접에서 설명할 수 있도록 선택 이유를 함께 정리한다.
- 작업 완료 후 다음 작업을 임의로 이어서 진행하지 않는다.
